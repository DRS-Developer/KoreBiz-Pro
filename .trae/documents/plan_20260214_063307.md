# Resposta: Desempenho do PWA Instalado

**Sim, o desempenho será o mesmo ou, na maioria dos casos, superior.**

Quando um usuário "instala" o PWA, ele não está baixando um software pesado separado. Ele está criando um atalho especial que roda o **mesmo código** do seu site, usando o **mesmo motor** do navegador (Chrome, Safari, Edge), mas com benefícios exclusivos:

1.  **Carregamento Instantâneo (Cache):** Para ser instalável, o PWA exige um *Service Worker*. Isso significa que, após a primeira visita, os arquivos do site (HTML, JS, CSS) ficam salvos no dispositivo. Ao abrir o app instalado, ele carrega do disco (0ms de latência), não da rede.
2.  **Mesma Performance de Renderização:** A velocidade de execução do React, animações e interações é idêntica à de abrir o site no navegador, pois o motor é o mesmo.
3.  **Foco Visual:** Ao rodar em modo "standalone" (sem barra de endereço/abas), a percepção de performance e imersão melhora, eliminando distrações visuais.

Portanto, transformar o site em PWA **não prejudica** o SEO ou a performance da versão web tradicional; pelo contrário, melhora a experiência para todos os usuários (instalados ou não) devido ao cache inteligente.

# Plano de Execução: Implementação PWA

Com essa confirmação, seguiremos com a transformação do projeto em PWA para habilitar atualizações automáticas e funcionamento offline.

### 1. Instalação e Configuração (`vite-plugin-pwa`)
*   **Ação:** Instalar o plugin e configurá-lo no `vite.config.ts`.
*   **Configuração Chave:**
    *   `registerType: 'prompt'`: O usuário decide quando atualizar (evita recargas inesperadas).
    *   `workbox.globPatterns`: Cachear JS, CSS, HTML, ICO, PNG, SVG.
    *   `devOptions.enabled: true`: Para testarmos o PWA em ambiente de desenvolvimento.

### 2. Componente de Notificação (`ReloadPrompt`)
*   **Ação:** Criar um componente visual (Toast/Banner) que aparece quando o Service Worker detecta uma nova versão no servidor.
*   **Fluxo:** O app verifica atualizações a cada X minutos ou na navegação. Se houver, exibe: *"Nova versão disponível. Clique para atualizar."*

### 3. Configuração do Manifesto (`manifest.webmanifest`)
*   **Ação:** Definir nome, cores, ícones e comportamento (`display: standalone`) para garantir que o app seja "instalável" e pareça nativo.

### 4. Integração com Offline-First
*   **Ação:** Garantir que o Service Worker não interfira nas requisições do Supabase (ignorar rotas de API na estratégia de cache estático), deixando o cache de dados para o seu sistema existente de IndexedDB.

### 5. Validação
*   Testar o fluxo de "Instalar" -> "Publicar nova versão" -> "Receber notificação" -> "Atualizar sem perda de dados".
